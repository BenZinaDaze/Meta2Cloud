# 115 网盘接入文档

[115 开放平台官方文档](https://www.yuque.com/115yun/open) 。

## 1. 接入概览

当前实现实际使用了 3 类外部地址：

### 1.1 认证相关

- `https://passportapi.115.com`
- `https://qrcodeapi.115.com`

### 1.2 业务接口

- `https://proapi.115.com`

### 1.3 上传对象存储

- 通过 `/open/upload/get_token` 返回的 `endpoint`
- 使用 STS 临时凭证访问
- 当前实现使用 `oss2` 做分片上传

## 2. 认证方式

115 开放平台支持两种授权方式：

| 模式 | 适用场景 | 说明 |
| --- | --- | --- |
| **PKCE 扫码模式** | 无后端的客户端应用 | 本文主要记录此模式 |
| **授权码模式** | 有后端服务的第三方应用 | 通过 `redirect_uri` 回调换取授权码 |

认证成功后需要持久化以下字段：

- `access_token`
- `refresh_token`
- `expires_in`
- `refresh_time`

后续访问 `https://proapi.115.com` 时，请求头需要带：

```http
Authorization: Bearer <access_token>
```

当前实现使用的 `client_id` 来自配置 `U115_APP_ID`，默认值 100197847

## 3. 认证接口

## 3.1 获取设备二维码

### 接口

- 方法：`POST`
- URL：`https://passportapi.115.com/open/authDeviceCode`

### 用途

生成二维码内容，以及后续轮询和换 token 所需的中间参数。

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `client_id` | `string` | 是 | 115 应用标识 |
| `code_challenge` | `string` | 是 | PKCE 挑战值 |
| `code_challenge_method` | `string` | 是 | 固定为 `sha256` |

### PKCE 生成规则

1. 本地生成 `code_verifier`（长度 43~128 位）
2. 对 `code_verifier` 做 `SHA256`
3. 将摘要做 Base64 编码，得到 `code_challenge`

当前实现还会把以下字段缓存到内存，供后续轮询和换 token 使用：

- `code_verifier`
- `uid`
- `time`
- `sign`

### 成功返回关键字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | `number` | 成功时为 `0` |
| `data.qrcode` | `string` | 二维码内容，直接展示给用户扫码 |
| `data.uid` | `string` | 设备登录 ID |
| `data.time` | `string/number` | 轮询状态需要 |
| `data.sign` | `string` | 轮询状态需要 |

### 示例

请求：

```http
POST /open/authDeviceCode
Content-Type: application/x-www-form-urlencoded

client_id=100197847&
code_challenge=xxx&
code_challenge_method=sha256
```

成功响应示意：

```json
{
  "code": 0,
  "data": {
    "qrcode": "https://...",
    "uid": "xxx",
    "time": "xxx",
    "sign": "xxx"
  }
}
```

## 3.4 刷新 token

### 接口

- 方法：`POST`
- URL：`https://passportapi.115.com/open/refreshToken`

### 用途

在 `access_token` 过期后刷新令牌。

> ⚠️ **请勿频繁刷新，否则会被列入频控。**

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `refresh_token` | `string` | 是 | 上一次登录或刷新返回的值 |

### 成功返回关键字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `state` | `number` | 状态 |
| `code` | `number` | 成功时为 `0` |
| `message` | `string` | 异常信息 |
| `data.access_token` | `string` | **新的**访问令牌，同时刷新有效期 |
| `data.refresh_token` | `string` | **新的**刷新令牌，**有效期不延长不改变** |
| `data.expires_in` | `number` | access_token 有效期秒数（默认约 `2592000` 即 30 天） |

> ⚠️ **重要**：调用 `/open/refreshToken` 后，`refresh_token` 本身也会重新生成。
> 必须将返回的新 `refresh_token` 更新到本地持久化存储，否则旧值会导致错误码 `40140120`。

### 推荐刷新逻辑

```text
if refresh_time + expires_in < 当前时间:
    调用 refreshToken
    更新 access_token / refresh_token / expires_in / refresh_time
```

## 3.5 认证时序

```text
客户端
  -> 生成 code_verifier / code_challenge
  -> POST /open/authDeviceCode
  <- qrcode, uid, time, sign

客户端展示二维码

客户端轮询
  -> GET /get/status/?uid=...&time=...&sign=...
  <- status

若 status == 2
  -> POST /open/deviceCodeToToken
  <- access_token, refresh_token, expires_in
```

## 4. 公共请求规则

所有业务接口都通过 `https://proapi.115.com` 访问。

### 4.1 请求头

当前实现固定使用：

```http
User-Agent: W115Storage/2.0
Accept-Encoding: gzip, deflate
Content-Type: application/x-www-form-urlencoded
Authorization: Bearer <access_token>
```

### 4.2 返回码处理

当前实现把以下返回码视为成功：

- `code == 0`
- `code == 20004`

其中 `20004` 在创建目录场景中被当作"目录已存在"。

### 4.3 错误与风控

当前实现有 4 类处理逻辑：

1. `HTTP 429`：进入全局冷却后重试
2. 返回 `message` 包含 `已达到当前访问上限`：进入全局冷却后重试
3. 其他 HTTP 错误：指数退避重试
4. 普通业务错误：直接返回失败

### 4.4 默认限流参数

当前实现写死了两组默认 QPS：

| 场景 | 默认值 | 说明 |
| --- | --- | --- |
| 普通业务接口 | `3 QPS` | 通过 `QpsRateLimiter(3)` 控制 |
| 下载地址接口 `/open/ufile/downurl` | `1 QPS` | 单独限流，避免下载链路过快触发风控 |

风控触发后的默认冷却时间：

| 场景 | 默认值 | 说明 |
| --- | --- | --- |
| `HTTP 429` 或返回 `已达到当前访问上限` | `3600 秒` | 即 `1 小时` 全局冷却 |

## 6. 下载接口

## 6.1 获取下载地址

### 接口

- 方法：`POST`
- URL：`https://proapi.115.com/open/ufile/downurl`

### 用途

根据文件的 `pick_code` 获取实际下载地址。

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `pick_code` | `string` | 是 | 文件详情或列目录结果里的 `pc/pick_code` |

### 返回值说明

当前实现没有按固定字段名取值，而是这样取：

```python
download_url = list(download_info.values())[0].get("url", {}).get("url")
```

这说明该接口返回结构可能不是常规的固定 `data.url`，你在自己程序里最好先打印原始响应确认。

## 6.2 实际文件下载

### 流程

1. 先调用 `/open/folder/get_info` 获取文件详情
2. 读取 `pick_code`
3. 再调用 `/open/ufile/downurl`
4. 拿到真实下载 URL 后直接流式下载

### 当前实现特点

- 使用 `httpx.stream("GET", download_url)` 直接下载
- 如果下载中断或异常，会删除本地残留文件

## 7. 上传接口

115 上传链路比普通网盘复杂，包含：

1. 文件特征值计算
2. 初始化上传
3. 二次认证
4. 秒传判断
5. 获取 STS 临时凭证
6. 查询断点续传信息
7. 对象存储分片上传
8. 上传完成回调

## 7.1 计算文件特征值

### 用途

上传前需要计算两个 SHA1：

| 字段 | 说明 |
| --- | --- |
| `fileid` | 整个文件的 SHA1 |
| `preid` | 文件前 `128MB` 内容的 SHA1 |

### 当前实现规则

- `fileid = sha1(整个文件)`
- `preid = sha1(前 128 * 1024 * 1024 字节)`

## 7.2 初始化上传

### 接口

- 方法：`POST`
- URL：`https://proapi.115.com/open/upload/init`

### 用途

告诉 115 准备上传一个文件，并返回秒传、二次认证、对象存储相关信息。

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `file_name` | `string` | 是 | 文件名 |
| `file_size` | `number` | 是 | 文件大小，单位字节 |
| `target` | `string` | 是 | 目标目录，当前实现格式为 `U_1_<cid>` |
| `fileid` | `string` | 是 | 全文件 SHA1 |
| `preid` | `string` | 是 | 前 128MB SHA1 |

### 成功返回关键字段

当前实现从 `data` 中读取：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `bucket` | `string` | 对象存储 bucket |
| `object` | `string` | 对象存储对象名 |
| `callback` | `object` | 上传完成回调配置 |
| `sign_check` | `string` | 二次认证范围，如 `2392148-2392298` |
| `pick_code` | `string` | 文件 pick_code |
| `sign_key` | `string` | 二次认证需要 |
| `code` | `number` | 业务状态码，`700/701` 触发二次认证 |
| `status` | `number` | `2` 表示秒传成功 |
| `file_id` | `string/number` | 秒传成功时可能返回 |

## 7.3 二次认证

### 触发条件

初始化上传返回：

- `data.code in [700, 701]`
- 并且存在 `sign_check`

### 处理逻辑

1. 解析 `sign_check`，格式如 `start-end`
2. 从本地文件中截取指定字节区间
3. 对该区间内容计算大写 SHA1，得到 `sign_val`
4. 再次调用 `/open/upload/init`

### 第二次请求新增参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `pick_code` | `string` | 是 | 第一次初始化返回 |
| `sign_key` | `string` | 是 | 第一次初始化返回 |
| `sign_val` | `string` | 是 | 按 `sign_check` 计算出的大写 SHA1 |

## 7.4 秒传

### 触发条件

初始化上传返回：

- `data.status == 2`

### 处理逻辑

当前实现认为文件已秒传成功，然后：

1. 读取 `file_id`
2. 再调用 `/open/folder/get_info`
3. 返回文件详情

## 7.7 对象存储分片上传

### 当前实现做法

1. 使用 `/open/upload/get_token` 返回的 STS 创建 `oss2.StsAuth`
2. 创建 `oss2.Bucket`
3. 以约 `10MB` 为偏好分片大小
4. 调用 `init_multipart_upload`
5. 逐片调用 `upload_part`
6. 最后调用 `complete_multipart_upload`

### 完成上传时附带的回调头

| 头 | 说明 |
| --- | --- |
| `X-oss-callback` | Base64 编码后的 `callback.callback` |
| `x-oss-callback-var` | Base64 编码后的 `callback.callback_var` |
| `x-oss-forbid-overwrite` | 当前实现固定为 `false` |

## 7.8 上传时序

```text
计算 fileid / preid
  -> POST /open/upload/init
  <- 初始化结果

若 code in [700, 701]
  -> 计算 sign_val
  -> POST /open/upload/init
  <- 二次认证结果

若 status == 2
  <- 秒传成功，结束

否则
  -> GET /open/upload/get_token
  <- endpoint + STS

  -> POST /open/upload/resume
  <- 续传信息

  -> OSS 分片上传
  -> complete_multipart_upload
  <- 成功
```

## 12.2 必须注意的实现细节

1. 上传不是普通表单上传，而是"115 接口 + 对象存储 STS"混合链路。
2. 下载依赖 `pick_code`，不要只靠文件路径设计下载逻辑。
3. 很多操作依赖先查询详情拿到 `file_id` 或 `pick_code`。
4. 115 有明显风控，生产环境一定要做限流、冷却和重试。当前实现默认是普通接口 `3 QPS`、下载地址接口 `1 QPS`、触发风控后冷却 `1 小时`。
5. `downurl` 的响应结构建议先记录原始 JSON，再按实际结果解析。
6. `code == 20004` 在当前实现里被当作成功，但你自己的程序最好按接口逐一验证语义。
7. 云下载任务的 `status` 字段：`-1` 失败，`0` 分配中，`1` 下载中，`2` 成功。
8. `refresh_token` 每次刷新后**值会变**，必须持久化新值。旧值不可复用。
